es 集群搭建

一、前言

本文章的目的是快速搭建一个高可用的es集群，本方案基于docker环境实现，本着快速、便捷、高可用的目的来搭建。

二、基本概念

2.1 集群（ cluster ）

集群是由一个或者多个拥有相同 cluster.name 配置的节点组成，共同承担数据和负载压力，当节点数量发生变化时集群将会重新平均分布所有数据。

2.2 节点（ node ）

一个运行中的 ES 实例称为一个节点

主节点负责管理集群范围内的所有变更，例如增加/删除索引，或者增加/删除节点等，且不需要涉及到文档级别的变更和搜索等操作
任何节点都能成为主节点
当集群只有一个主节点，即使流量增加也不会成为瓶颈

2.3 索引（ index ）

名词；类似于传统关系数据库中的一个数据库
动词；索引一个文档就是存储一个文档到一个索引(名词)中以便它可以被检索和查询到。类似于 SQL 语句中的 INSERT 关键词
倒排索引；类似于传统关系型数据库中的索引概念，可以提升数据检索速度

2.4 类型（ type ）

一个索引包含一个或多个 type ，相当于传统关系型数据库中的表

2.5 文档（ document ）

相当于传统关系型数据库中的数据行

2.6 分片（ shards ）

是一个底层的「工作单元」，仅保存了全部数据的一部分
是数据的容器，文档保存在分片内，分片又被分配到集群内的各个节点里
当集群规模扩大或者缩小时， ES 会自动的在各节点中迁移分片，使得数据仍然均匀分布在集群里
分为「主分片」和「副本分片」
在索引建立的时候就已经确定了主分片数，但是副本分片数可以随时修改；默认情况下会被分配「 5 」个主分片和「 1 」份副本（每个主分片拥有一个副本分片）
相同主分片的副本分片不会放在同一个节点
① 主分片 ( Primary shard )

索引内任意一个文档都归属于一个主分片，所以主分片的数目决定着索引能够保存的最大数据量

② 副本分片（ Replica shard ）

只是一个主分片的拷贝，作为硬件故障时保护数据不丢失的冗余备份，并为搜索和返回文档等读操作提供服务

四、安装elasticsearch-head插件

elasticsearch-head是一个可视化的es集群管理页面。

4.1 安装elasticsearch-head为插件

  • Elasticsearch 5.x，6.x和7.x 不在支持插件模式安装，需要作为一个单独的服务运行。
  • Elasticsearch 2.x 版本，可以直接安装为插件 sudo elasticsearch/bin/plugin install mobz/elasticsearch-head
  • for Elasticsearch 1.x 版本，可以直接安装为插件 sudo elasticsearch/bin/plugin -install mobz/elasticsearch-head/1.x
  • for Elasticsearch 0.x 版本，可以直接安装为插件 sudo elasticsearch/bin/plugin -install mobz/elasticsearch-head/0.9
  • 打开 http://localhost:9200/_plugin/head/ 即可使用

这将自动从github下载合适版本的elasticsearch-head并将其作为插件运行在elasticsearch集群中。 在这种模式下，elasticsearch-head自动连接到运行它的节点

4.2 作为独立服务运行

  • git clone git://github.com/mobz/elasticsearch-head.git
  • cd elasticsearch-head
  • npm install
  • npm run start
  • open http://localhost:9100/

这将在端口9100启动一个本地Web服务器，该端口提供elasticsearch服务
